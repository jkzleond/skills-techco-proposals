## AI 交互流程

当使用本 skill 时，AI Agent 应按以下流程与用户交互：

### 步骤1：检查用户配置

**读取用户配置文件：** `.claude/user-config.yaml`

如果配置文件存在且包含用户信息：
- 使用配置中的公司名称
- 跳过询问步骤

如果配置文件不存在或用户信息为空：
- 使用 `AskUserQuestion` 工具询问用户信息
- 询问问题：
  1. 您的姓名？（可选）
  2. 您的公司名称？
  3. 您在公司担任什么角色？
- 将用户信息保存到 `.claude/user-config.yaml`
- 格式如下：
  ```yaml
  user:
    name: "张三"
    company_name: "某某科技有限公司"
    company_role: "售前产品经理"

  last_updated: "2026-02-04"
  ```

### 步骤2：编制方案

使用用户配置中的 `company_name` 作为编制单位，按照SKILL的规范编制方案。

### 步骤3：输出方案

确保方案文档的头部信息正确：
```markdown
# [项目名称]建设方案

**编制单位：** [从配置读取的公司名称]
**编制日期：** 2026年2月
**版本号：** v1.0
```

**⚠️ 占位符处理：**

本SKILL包含占位符 `{{COMPANY_NAME}}`，AI Agent需要：

1. **读取用户配置**：从 `.claude/user-config.yaml` 读取 `company_name`
2. **全局替换**：在执行时将所有 `{{COMPANY_NAME}}` 替换为用户的公司名称
3. **输出文档**：确保输出的文档中使用正确的公司名称

如果用户配置为空或不存在：
- 首次使用时询问用户（见步骤1）
- 或使用默认值"您的公司名称"

## 方案结构要求

### 一、项目背景与建设目标

**包含内容：**
- 现状分析（当前情况、存在的问题）
- 会员/客户的真实反馈（使用引用增强说服力）
- 解决方案思路（用流程图展示，⭐ **必须**使用 ` ```ascii:flowchart ` `）
- 建设目标（用表格呈现）

**⚠️ ASCII图规范提醒：**
- ✅ 所有流程图、架构图都必须标注类型
- ✅ 流程图用 ` ```ascii:flowchart ` `
- ✅ 如果涉及界面展示，用 ` ```ascii:ui ` `

**目录标题：** 使用商务化表述
- ✅ "一、项目背景与建设目标"
- ❌ "一、为什么要做"

**二级标题：** 使用描述性标题
- ✅ "1.1 现状分析"
- ✅ "1.2 会员的真实反馈"
- ✅ "1.3 解决方案：XXX"

### 二、功能方案设计

**包含内容：**
- 系统定位（是什么、不是什么）
- 系统架构（用ASCII图展示，⭐ **必须**使用 ` ```ascii:architecture ` `）
- 功能设计（图文并茂，用表格、示例、流程图）
- 与现有系统的关系（用架构图说明，⭐ **必须**使用 ` ```ascii:architecture ` `）

**⚠️ ASCII图规范提醒：**
- ✅ 所有系统架构图、流程图、UI界面图、时间线图都必须标注类型
- ✅ 架构图用 ` ```ascii:architecture ` `
- ✅ 流程图用 ` ```ascii:flowchart ` `
- ✅ UI界面用 ` ```ascii:ui ` `
- ✅ 时间线用 ` ```ascii:timeline ` `
- ✅ 其他示意图用 ` ```ascii:diagram ` `

**二级标题示例：**
- ✅ "2.1 系统架构：不折腾，简单好用"
- ✅ "2.2 会员端功能：简单直观"
- ✅ "2.3 管理端功能：自主可控"

### 三、投资预算

**包含内容：**
- 开发费用（明确数字，中文大写）
- 费用构成（用表格展示功能模块，让客户看到价值）⭐
- 费用包含内容（按阶段展开工作量）
- 费用不包含内容（表格形式）
- 付款方式（用流程图展示，⭐ ASCII图必须使用代码块，语言标注为 `ascii`）

**推荐方式：混合拆分（模块 + 阶段）**

先展示功能模块表格，让客户看到"钱花在哪些功能上"；
再展开开发阶段，让客户了解"我们会做这么多事"。

**费用构成表格示例：**

**费用包含内容示例：**

整个项目按以下阶段实施：

#### 需求分析与设计阶段
- [具体工作内容1]
- [具体工作内容2]

#### 功能开发阶段

  - [功能1]
  - [功能2]

### 四、实施周期

**包含内容：**

- 实施阶段（分阶段详细说明）
- 交付物清单（分类列出）

## 图文并茂写作规范

### 使用ASCII图表

**⚠️⚠️⚠️ 关键规范：每一个ASCII图都必须标注类型 ⚠️⚠️⚠️**

**❌ 错误示例（不要这样写）：**
```markdown
```
┌─────────────────┐
│   用户界面       │
└─────────────────┘
```
```

**问题：** 没有标注类型，converting-markdown skill 无法识别，转换时无法生成精美图形

**✅ 正确示例（必须这样写）：**
```markdown
```ascii:ui
┌─────────────────┐
│   用户界面       │
└─────────────────┘
```
```

**要求：** 每一个、每一个、每一个 ASCII 图都必须标注类型（architecture/flowchart/ui/timeline/diagram）

---

**支持的图形类型：**
所有ASCII图必须使用代码块格式，并从以下 5 种类型中选择最合适的标注：

````markdown
```ascii:architecture
┌─────────────────┐
│   系统架构图     │
└─────────────────┘
```

```ascii:flowchart
步骤1 → 步骤2 → 步骤3
```

```ascii:ui
┌─────────────────┐
│   用户界面       │
└─────────────────┘
```

```ascii:timeline
Week 1    Week 2    Week 3
```
````

**支持的图形类型：**
- `architecture` - 系统架构图（包含嵌套方框、层级关系）
- `flowchart` - 流程图（箭头连接的步骤）
- `ui` - UI界面图（按钮、输入框、布局）
- `timeline` - 时间线图（甘特图、进度图）
- `diagram` - 通用图（其他类型的示意图）

**为什么必须使用代码块并标注类型：**
1. ✅ **保持等宽字体，确保字符对齐** - 避免Markdown渲染时格式错乱
2. ✅ **智能转换SVG** - converting-markdown skill根据类型选择最优转换策略
3. ✅ **精准识别** - 不会把普通文本中的框线字符误认为ASCII图
4. ✅ **版本控制友好** - git diff更清晰，代码块格式更易于比较

**什么时候用ASCII图：**
- 展示系统架构
- 展示界面布局
- 展示数据流程
- 展示时间进度

**示例：**

**系统架构图（architecture）：**
````markdown
```ascii:architecture
┌─────────────────┐
│   现有系统       │
│                 │
└─────────────────┘
    ↓ 只读访问
┌─────────────────┐
│   新增系统       │
│                 │
└─────────────────┘
```
````

**流程图（flowchart）：**
````markdown
```ascii:flowchart
步骤1 → 步骤2 → 步骤3
  ↓       ↓       ↓
 结果1   结果2   结果3
```
````

**UI界面图（ui）：**
````markdown
```ascii:ui
┌────────────────────────────────────────┐
│       我的积分                         │
│        1,000  积分                     │
│  [去商城兑换]  [查看明细]              │
└────────────────────────────────────────┘
```
````

**时间线图（timeline）：**
````markdown
```ascii:timeline
Week 1    Week 2    Week 3
│         │         │
│ 需求     │ 开发    │ 上线
└─────────┴─────────┘
```
````

**通用图（diagram）：**
````markdown
```ascii:diagram
┌─────────────────┐     ┌──────────┐
│   层级1         │ ──→ │   层级2  │
└─────────────────┘     └──────────┘
```
````

说明：当图形不属于上述4种类型时，使用 `diagram` 作为通用类型。

## 语言风格要求

### 使用流畅的叙述

❌ **错误（清单式）：**
```markdown
### 当前问题

1. 系统分散
2. 智能化服务缺失

✅ **正确（叙述式）：**

协会目前运营两个独立系统：官网和业务系统。
这导致会员需要在两个系统间频繁切换，
操作流程繁琐，用户体验不佳。

其次，缺少智能化服务工具。
会员咨询依赖人工，常见问题重复解答，
人力成本高，服务效率低。
```

❌ **错误（抽象描述）：**
```markdown
会员可以通过商城兑换商品。
```

✅ **正确（具体场景）：**
```markdown
**兑换流程：**

1. 点击"立即兑换"
      ↓
2. 确认兑换（检查积分余额）
      ↓
3. 提交订单
      ↓

**让积分真正用起来、有价值**

目前会员累计了积分，但缺少消费场景，

积分商城为会员提供积分兑换平台，
